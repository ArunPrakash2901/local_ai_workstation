# R12: Apply-Ready Checker Implementation

## Summary
The read-only supervised apply readiness checker has been implemented as `ws apply-ready`. This command serves as the final strict gatekeeper before an operator executes `ws agent-run`, verifying all context conditions, repo state, canary status, and the presence of a localized plan without initiating codebase mutation.

## Files Changed
- `scripts/ws`: Added `apply-ready` to the help menu and the command dispatcher right before `agent-run`.
- `scripts/ws_apply_ready.sh`: A new bash script implementing the preflight execution and boundary checks.
- `.gitignore`: Updated to explicitly ignore the transient `reports/APPLY_READY_*.md` diagnostic reports.
- `WORKSTATION_MANUAL.md`: Updated to instruct the operator to use `ws apply-ready` as the final read-only confirmation step when reviewing a local loop.

## Checks Implemented
The new `ws apply-ready` script actively evaluates:
1.  **Project Existence:** Verifies project key via `projects.yaml`.
2.  **Repository Path:** Validates the physical repository existence.
3.  **Task File:** Confirms task file exists via the `to_wsl_path` translation.
4.  **Task Boundaries:** Reads the task file for explicit `Allowed Files:` restrictions.
5.  **Clean Repository:** Runs strict `git diff --porcelain` to block any runs on a dirty working tree.
6.  **Canary Status:** Reads `reports/agent_canary_status.json` ensuring `AGENT_CANARY_PASSED` is the current status.
7.  **Stale Runs:** Evaluates `auto_runs/` for active `CODEX_RUNNING` locks.
8.  **Local Plan Evidence:** Verifies that a `local_plan.md` artifact recently generated by `ws loop-start` exists to confirm local planning occurred.

If all pass, it assigns the classification `APPLY_READY` and outputs the exact `ws agent-run` command syntax. If any fail, it maps to respective blocked terminal states (e.g., `BLOCKED_DIRTY_REPO`, `BLOCKED_CANARY_FAILED`).

## Validation Run
Executing `ws apply-ready` accurately detected uncommitted changes on the `workstation_control_plane` repo, successfully aborting the check with `BLOCKED_DIRTY_REPO` while surfacing a clean `APPLY_READY_*.md` diagnostic report safely ignored by git tracking.

## Limitations
- State evaluation checks active folder structures rather than a synchronized locking service.
- Requires manual invocation rather than being implicitly chained to `ws loop-start` (by design for supervised workflow isolation).

## Next Steps
The workstation now has fully verified read-only planners, explicit local-first isolated loops, diagnostic dashboards, and a strict preflight mechanism for manual cloud apply handoffs. The implementation of robust automation is fully grounded by these preflight invariants.
