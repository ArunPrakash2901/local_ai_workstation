# R8: Supervised Single-Loop Start Design
**Date:** 2026-05-16
**Status:** Design Only (Implementation Deferred)

This document outlines the architecture for the first minimal viable product (MVP) of `ws loop-start`, a supervised command that initiates a single task loop.

## 1. Scope of the First MVP
**What should `ws loop-start` do?**
In its first iteration, `ws loop-start` will act as a strict orchestrator that consumes a `ws loop-plan` classification, verifies preflight health, and executes the designated bounded workflow for a single project and task. It will run in the foreground (supervised) and will not detach into a background process.

**Should the first implementation support only local-plan mode?**
Yes. To guarantee safety and stability during the initial rollout, the MVP will only execute loops classified as `LOCAL_PLAN_ONLY` (effectively wrapping `ws build --plan-only`). 

**Should cloud-apply mode remain disabled?**
Yes. Cloud-apply loops (which wrap `ws agent-run`) will remain disabled in `ws loop-start` until the Codex canary is consistently passing and the local-plan loop orchestration has been proven flawless. If a task is eligible for cloud apply, `loop-start` will currently gracefully degrade it to a manual handoff recommendation.

## 2. Orchestration and Preflight
**How should loop-start consume the latest ws loop-plan result?**
`ws loop-start <project_key> <task_file>` will inherently run `ws loop-plan` dynamically in memory to get the freshest classification. It will not rely on parsing old `LOOP_PLAN_*.md` files to avoid race conditions.

**What preflight checks must be repeated before starting?**
The dynamic `loop-plan` invocation will automatically re-verify:
- Project repository paths.
- Task file existence and `Allowed Files` boundaries.
- Git repository cleanliness.
- Cloud canary status.

## 3. State Management and Outputs
**What terminal states should loop-start produce?**
- `LOCAL_PLAN_COMPLETED`: Successfully executed the local plan.
- `HANDOFF_READY`: Task requires manual cloud apply or human intervention.
- `LOOP_BLOCKED`: Preflight checks failed (e.g., dirty repo, missing files).

**What files/reports should it write?**
- A transient `reports/LOOP_START_<timestamp>.md` logging the initiation, preflight results, and termination state.
- The standard artifacts generated by `ws build --plan-only` (e.g., `local_plan.md` inside `build_runs/`).

**How should it avoid duplicate starts?**
Before starting, it will check the `build_runs/` and `auto_runs/` directories for recent, active folders matching the target project and task slug. If a run is currently active, it will block execution.

**What should happen when the repo is dirty?**
The internal `loop-plan` check will classify it as `BLOCKED_DIRTY_REPO`, and `loop-start` will immediately exit with `LOOP_BLOCKED`, refusing to execute.

**What should happen when cloud quota is exhausted?**
The internal `loop-plan` check will classify it as `BLOCKED_CLOUD_QUOTA`. The `loop-start` command will exit gracefully, outputting `HANDOFF_READY` and instructing the operator to use `ws agent-run ... --mode detect`.

## 4. Deferred Complexity
**Why do worktree creation and parallel execution remain out of scope?**
Worktrees introduce complex state management (e.g., orphaned worktrees, detached HEADs, locked index files). Parallel execution introduces race conditions around model VRAM allocation and registry locks. The supervised MVP must first prove that it can flawlessly orchestrate sequential, single-repository tasks on the primary branch before introducing multi-tenant complexity.

## 5. Required Implementation Changes (For the future)
To implement this design, the following files will eventually need to be modified or created:
1. `scripts/ws`: Add `loop-start` to dispatcher.
2. `scripts/ws_loop_start.sh`: Create the core orchestrator script.
3. `.gitignore`: Ignore `reports/LOOP_START_*.md`.
4. `WORKSTATION_MANUAL.md`: Document usage.
