# Learning State Synchronization Apply v1

## Purpose
This document defines the guarded state synchronization apply layer for Learning Strongholds (Phase 7B). This layer is the first phase where `state.json` can be modified, and it follows an extremely conservative implementation to ensure system integrity.

## Phase Boundary
- **Phase 7A**: Learning State Sync Planner v1 (Dry-run only).
- **Phase 7B**: Guarded Learning State Sync Apply v1 (Guarded mutation).

## Command Shape
The primary interface is through the `ws` unified command.

### Dry-Run Preview
```bash
ws learning-state-sync-apply <stronghold_id> --dry-run
ws learning-state-sync-apply <stronghold_id> --dry-run --json
```

### Actual Apply
```bash
ws learning-state-sync-apply <stronghold_id> --confirm-sync
ws learning-state-sync-apply <stronghold_id> --confirm-sync --json
```

## Eligibility Rules
A proposed state change can be applied only if ALL of the following are true:
1. **Source Plan**: Generated by `learning_state_sync_planner_v1`.
2. **Risk Level**: Must be `LOW`.
3. **Evidence Quality**: Must be `strong`.
4. **Phase 7B Eligibility**: `apply_allowed_in_phase_7b` must be `true`.
5. **Schema Allowlist**: Target path must be in the `STATE_PATH_ALLOWLIST`.
6. **Artifact Verification**: The source artifact referenced in the confirmation must exist on disk and be within the stronghold.
7. **State Freshness**: The current value in `state.json` must match the `current_value` recorded in the plan at the time of planning.

## Blocked Changes (v1)
The following changes are explicitly blocked from automated apply in Phase 7B:
- `current_state` changes to `READY_FOR_ADVANCEMENT` (HIGH risk).
- Any advancement-stage mutation.
- Any lesson completion mutation.
- Any review schedule mutation (unless explicitly LOW risk and planner-approved).
- Medium or High risk changes.
- Stale artifact deletion/archive/move.

**Advancement remains a manual human operation in this phase.**

## Backup Model
Before any mutation to `state.json`, a timestamped backup is created:
`strongholds/learning/<stronghold_id>/state_backups/state_YYYYMMDDTHHMMSSZ_before_sync.json`

## Audit Ledger Model
Every successful apply appends an audit record to:
`strongholds/learning/<stronghold_id>/state_sync_audit.jsonl`

Audit record includes:
- `sync_id`, `timestamp_utc`, `stronghold_id`
- `planner_plan_id`
- `applied_changes` (list of change objects)
- `skipped_changes`, `blocked_changes`
- `backup_path`

## Rollback Procedure (v1)
In Phase 7B, rollback is a manual operator process:
1. Locate the backup file in `strongholds/learning/<stronghold_id>/state_backups/`.
2. The exact backup path is recorded in the `state_sync_audit.jsonl` entry for the sync.
3. Verify the backup file exists and is valid JSON.
4. Copy the backup file over `state.json`.
5. Append a manual record to `state_sync_audit.jsonl` noting the rollback.

## Validation Results
Validation was performed using `_test_isolation_fixture`.

### Tests Run:
1. **Refusal without mode**: PASS.
2. **Refusal with conflicting modes**: PASS.
3. **JSON Error Purity**: PASS (Verified that errors in `--json` mode are returned as valid JSON).
4. **Path Traversal Guard**: PASS (Verified that path traversal attempts are blocked).
5. **Dry-run non-mutation**: Verified `state.json`, backups, and audit ledger were untouched.
6. **Confirm-sync implementation**:
   - Verified 3 LOW-risk changes were applied correctly.
   - Verified HIGH-risk advancement was blocked.
   - Verified backup was created before write.
   - Verified audit record was appended after write.
   - Verified atomic-ish write (temp file + rename).
   - Verified post-write state integrity.

### Cross-OS Stability:
Path normalization was implemented in both the planner and the applier to handle Windows (`D:\...`) and WSL (`/mnt/d/...`) path differences in the confirmation ledger.

## Known Limitations
- Advancement to `READY_FOR_ADVANCEMENT` is blocked (by design).
- `next_learning_task` updates are currently marked MEDIUM risk by the planner and thus blocked.
- Only exact string matches for current values are used for freshness checks.
