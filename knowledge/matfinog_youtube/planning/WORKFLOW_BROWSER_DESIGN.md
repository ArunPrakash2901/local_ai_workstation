# Workflow Browser Design

## Purpose
The Workflow Browser is a read-only interface designed to allow the human researcher to explore the MatFinOg YouTube corpus via structured workflows and safe research prompts. It serves as the primary navigation tool for the Quant Research Workflow Coach, bridging the gap between passive learning and structured quantitative research without generating automated trading logic.

## Target User
The target user is the human quantitative researcher/developer who uses this tool to identify safe research pathways, validation checks, and risk frameworks before writing code or running backtests.

## Non-Goals
- Generating trading signals, financial advice, or investment recommendations.
- Interacting with live brokers or generating automated execution bots.
- Running active backtests within the browser itself.
- Summarizing entire transcripts using LLMs (relying instead on deterministic snippets).

## Core Screens/Views

### 1. Workflow Overview
A dashboard displaying the 8 canonical workflows (e.g., `risk_first_strategy_review_workflow`, `market_inefficiency_hypothesis_workflow`). It shows the total count of videos and available research prompts for each workflow.

### 2. Topic Overview
A view that clusters the library by supporting topics (e.g., `risk_management`, `execution_and_microstructure`), allowing users to browse horizontally across different workflows.

### 3. Prompt Library
The main table/list view of the 366 structured research prompts.
- **Columns:** Prompt Title, Type, Workflow, Source Video Title, Suggested Next Human Action.

### 4. Source Video Detail
A drill-down view showing a specific source video, its URL, the associated workflow, and all generated prompts linked to it. It includes the verbatim `evidence_snippet` to maintain source grounding.

### 5. Human Review Queue
A view filtering prompts where `human_review_required` is `true`. This acts as a task list for the researcher to validate concepts manually before proceeding.

### 6. Research Backlog Candidate View
A staging area where a user can "pin" or select specific prompts (e.g., a combination of a hypothesis prompt and a risk review prompt) to form a complete research plan.

### 7. Safety / Forbidden Use View
A persistent informational view explaining the safety constraints, reminding the user that no financial advice or live trading code will be generated.

## Features

### Filters
- By `source_workflow_id`
- By `prompt_type` (e.g., `validation_question`, `risk_review_question`)
- By `source_topic_ids`

### Sorting Options
- By `processed_at` (newest/oldest)
- By `source_video_id`
- By prompt count (in aggregated views)

### Source Traceability View
Every prompt will have a clickable link or modal displaying the `evidence_snippet` and a direct link (`source_url`) to the exact video timestamp or page, ensuring the user can always verify the original context.

### Safety Banner
A persistent banner across all views explicitly stating: "SAFETY GATES ACTIVE: No financial advice, trading signals, or bot logic is generated by this application."

## Human Review Workflow
1. User browses to a specific prompt (e.g., "What specific risk checks should be defined?").
2. User clicks to view the `evidence_snippet` and original video.
3. User reads the `suggested_next_human_action` (e.g., "Identify required data before requesting implementation.").
4. User manually drafts their interpretation and selects "Export to Research Notebook" (Future feature).

## Future Integration
- **WS Command Surface:** Future integration could allow the browser to be launched via a CLI command (e.g., `ws matfinog browser`).
- **Research Notebook Export:** The browser will eventually support exporting selected prompts into a sterile Jupyter Notebook template (Milestone 3C), which enforces in-sample/out-of-sample data splits.
- **UI Options:** Future implementation may use lightweight, local-first frameworks like Textual (TUI) or a simple local HTML/JS interface without external dependencies.
