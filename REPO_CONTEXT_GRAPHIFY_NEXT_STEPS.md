# Repo Context & Graphify Next-Step Alignment Plan

## 1. Purpose
Graphify/Repo Context exists to reduce token usage by generating compact repo-intelligence artifacts. By using a pre-computed graph of the repository, local models and agents can navigate the codebase using high-signal summaries instead of expensive full-repo scans.

## 2. Current Boundary
- **Artifacts vs. Source:** Graphify outputs (`graph.json`, summaries, reports) are transient artifacts and derived intelligence. The repository source code remains the absolute source of truth.
- **Bounded Context:** Context packets generated by the Repo Context Lane are bounded summaries specifically tailored for consumption by agents and local models (Ollama/Hermes 3 8B).

## 3. Intended Pipeline
The Repo Context Lane follows a strict, human-gated pipeline to ensure safety and token efficiency:
1. **Project Path Identification:** Identify the target project root (e.g., `D:\portfolio_website`).
2. **Safe Inventory:** Shallow scan of directory structure and file metadata.
3. **Graphify Plan:** Generate a specific Graphify command plan with safety filters.
4. **Graphify Run:** Execute the approved Graphify command to generate `graph.json`.
5. **Graph Output Intake:** Verify and process the run results.
6. **Graph Summary:** Distill the heavyweight graph into a compact, readable Markdown summary.
7. **Task-Specific Context Packet:** Combine inventory and graph summaries for a specific task.
8. **Exchange/Runtime Handoff:** Package the context into a prompt for a downstream agent.

## 4. Safety Boundaries
- **Isolated Execution:** Never run Graphify on the workstation root (`D:\_ai_brain`) or the entire drive (`D:\`).
- **External Outputs:** Never write `graphify-out` inside the source repository. All outputs must reside in the centralized results directory: `D:\graphify-results\<project>\`.
- **Exclusion Policy:** Automatically ignore `.git`, `node_modules`, `dist`, `build`, caches, `.env`/secrets, and heavyweight data/media folders.
- **No Side Effects:** Graphify runs must not execute application code, modify source files, or perform git operations (commits/pushes).

## 6. Suggested Next Commands (Documentation Only)
The following commands represent the intended workflow. They should be run with `--dry-run` or as documented below:

- **Inventory:** `ws repo-context inventory --project <path> --dry-run`
- **Plan Generation:** `ws repo-context graphify-plan --project <path> --output <path> --dry-run`
- **Plan Review & Run:** `ws repo-context graphify-run --plan <plan_id> --confirm`
- **Intake:** `ws repo-context graphify-intake --run <run_id> --dry-run`
- **Packet Generation:** `ws repo-context packet --project <id> --task <task_name> --dry-run`

## 7. Short Slash Commands
Proposed slash commands for quick operator access:
- `/repo`: Show repository context status.
- `/context`: View/list generated context packets.
- `/graph`: Show Graphify run status and plan list.
- `/map`: Display the compact repo map summary.

## 8. How this helps Local Models
- **Token Compression:** Smaller context packets allow local models with limited context windows to "understand" large repos.
- **Reduced Latency:** Fewer full-repo reads mean faster response times.
- **Precision Scoping:** Context is bounded by the specific task, reducing noise and "hallucination" risk.
- **Safety:** Bounded local model prompts prevent accidental leak of sensitive repo data to cloud providers.

## 9. Next Implementation Recommendation
The next 2–3 slices for the Repo Context Lane should focus on:
1. **Graphify Output Intake/Report:** Hardening the verification of `graph.json` generation and logging success/failure metadata.
2. **Graph Summary:** Implementing the logic to distill `graph.json` AST/Community nodes into the standard `GRAPH_REPORT.md` format.
3. **Task-Specific Context Packet:** Automated assembly of the final prompt artifact for the Exchange Lane.
