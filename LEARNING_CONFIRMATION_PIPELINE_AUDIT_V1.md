# Learning Confirmation Pipeline Audit v1

## Purpose
This audit evaluates the stability, safety, and deterministic behavior of the Learning Confirmation pipeline (Phases 1-6) before implementing Phase 7 (State Synchronization).

## Audited Components
- **Action Generation**: `scripts/learning_action_pack.py`
- **Confirmation Core**: `scripts/learning_confirmation_core.py`
- **Command Wrappers**: `scripts/ws`, `scripts/ws.ps1`, `scripts/ws_learning_*.sh`
- **TUI Dashboard**: `tui/app.py` (Plain mode)
- **Validation Suite**: `scripts/test_learning_*.py`

## Audit Results

### 1. Command Surface Status: STABLE
- All standard commands documented in `LEARNING_CONFIRMATION_TUI_APPLY_V2.md` and `LEARNING_CONFIRMATION_CORE_V1.md` are verified working.
- PowerShell wrapper `ws.ps1` correctly delegates to WSL bash scripts.
- Argument passing is consistent across all layers.

### 2. JSON Contract Status: VERIFIED
- Both `learning_action_pack.py` and `learning_confirmation_core.py` return pure JSON when `--json` is specified.
- No human-readable banners or ANSI formatting leak into the JSON stream.

### 3. Action ID Stability: DETERMINISTIC
- **Fix Applied**: Action IDs were previously timestamp-based (changing every 24 hours). They are now deterministic, generated by hashing `stronghold_id`, `action_type`, and `evidence`.
- This ensures that duplicate protection remains robust even if an action is proposed over multiple days.

### 4. Duplicate Protection Status: ROBUST
- **Enhancement**: The CLI confirmation core now scans the ledger for existing `original_action_id` entries with `CONFIRMED_APPLIED` status before processing a confirmation.
- The TUI also performs this check before offering the `[a] Apply` option.

### 5. Ledger Integrity Status: VERIFIED
- The ledger (`learning_confirmations.jsonl`) is append-only.
- **Enhancement**: The `artifact_path` is now included directly in the ledger record for better auditability.
- Malformed lines in the ledger are handled gracefully by the TUI discovery logic.

### 6. Artifact Integrity Status: VERIFIED
- Confirmed artifacts are written to `confirmed_actions/` with unique timestamped filenames.
- Verification logic confirms that exactly one artifact is created per apply event.

### 7. state.json Protection Status: SECURE
- Verified that no script in the confirmation pipeline writes to `state.json`.
- TUI apply flow verifies `state.json` modification time (mtime) to ensure no accidental mutation occurred.

### 8. TUI Safety Status: SECURE
- Hard guards in `run_learning_confirmation_command` block mixed modes (`--dry-run` and `--confirm`) and shell metacharacters.
- Two-step typed confirmation (`APPLY <ACTION_ID>`) prevents accidental triggers.
- Subprocess execution uses argument lists (no `shell=True`) and has a 15-second timeout.

### 9. Test Isolation Status: ISOLATED
- **Fix Applied**: All write-heavy tests have been migrated to use `strongholds/learning/_test_isolation_fixture/`.
- The live `fine-tuning-small-open-source-models` stronghold is no longer polluted by automated validation runs.

## Remaining Limitations
- **Manual State Sync**: Confirmation records exist in the ledger but do not yet update the primary `state.json` (Phase 7).
- **Textual Mode**: Confirmation interactivity is currently limited to `plain` mode; Textual mode remains in a read-only state.

## Conclusion
The Learning Confirmation pipeline is **STABLE** and **READY** for Phase 7 (State Synchronization). The deterministic action IDs and reinforced duplicate protection provide a solid foundation for controlled learner advancement.
